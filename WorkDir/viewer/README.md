# ALDA Placement Viewer

A KLayout-style desktop viewer for ALDA placement JSON files, with built-in DRC spacing checker for both GUI and CLI use.

---

## Requirements

```
pip install PyQt6
```

---

## Starting the app

From anywhere in the project:

```bash
# Open empty viewer
python WorkDir/viewer/main.py

# Open with a file immediately
python WorkDir/viewer/main.py --input WorkDir/json_files/s42_n10_py001_v01.json
```

The `--input` path is resolved relative to your CWD, then relative to `WorkDir/`, then relative to the project root — so the above works regardless of which directory you run from.

> **Legacy:** a bare positional argument still works (`main.py <file.json>`) for backward compatibility.

---

## CLI DRC mode

When `--drc` is supplied the app never opens a window. The placement is checked against the PDK rules and results are printed to stdout.

```bash
python WorkDir/viewer/main.py \
    --input WorkDir/json_files/s13_n7_py101_v01.json \
    --drc   WorkDir/myPDK/gpdk090_device_rules.json
```

### Arguments

| Argument | Required | Description |
|----------|----------|-------------|
| `--input <JSON>` | yes (CLI) | Placement JSON file to check |
| `--drc <PDK_JSON>` | — | PDK rules file; enables CLI mode when present |

### Exit codes

| Code | Meaning |
|------|---------|
| `0` | DRC clean — no violations found |
| `1` | One or more DRC violations found |
| `2` | Error — bad path, invalid PDK file, or file has no placement result |

### Example output

```
Input   : …/json_files/s13_n7_py101_v01.json
PDK     : …/myPDK/gpdk090_device_rules.json
Blocks  : 7 placed

DRC result: 10 violation(s) found
------------------------------------------------------------------------
  B0 <-> B1: different_bulks (active)  required 2.000 µm,  actual 1.000 µm
  B0 <-> B2: different_bulks (poly)    required 3.000 µm,  actual 1.500 µm
  …
------------------------------------------------------------------------
Summary:
    5  different_bulks
    5  different_deviceGroups
```

## File format

The viewer loads `s<seed>_n<blocks>_py<stage>_v<version>.json` files generated by the ALDA pipeline.  
Both direct-dict format (from `main.py`) and n8n list-wrapped format (from `testFiles/`) are supported.

**Placement result** (stage `py101`) is required for DRC checks and Placement mode. Browser mode works with any stage.

## Tabs

**Browser mode:** each block gets its own tab (`B0 nmos1v`, `B1 nmos1v`, …). Click a tab to inspect that block.

**Placement mode:** a single **Placement** tab shows all blocks. Right-click any block to open its blueprint in a floating window.

---

## Block info panel (right)

- **Variant picker** — switch between layout variants; canvas updates and re-fits automatically.
- **Pins** — lists each pin, the net it connects to, and which other blocks share that net.
- **Grid** — set the main step and small step (in µm); grid starts at the scene origin `(0, 0)`.

In Placement mode the right panel shows **Placement info**: best-run metrics, an all-runs table, and the selected block's details.

---

## Layers panel (left)

Toggle visibility of:

| Layer | What it shows |
|-------|---------------|
| Block outline (Annotation) | `main_bbox` rectangles, colored by device type |
| Power nets (VDD/VSS) | Net stubs for VDD (red) and VSS (blue) |
| Signal nets | Net stubs for signal connections (green) |
| Labels | Pin names and net labels |
| Symmetry pairs | Symmetry group overlays |
| **DRC violations** | DRC error polygons drawn by the checker |

The **DRC violations** layer lets you hide all error highlights at once without closing the DRC window.

## Net stubs

Each pin draws a colored line from the block center outward to the edge and beyond.  
At the stub end: `P0 → net_id` with connected block IDs in parentheses, e.g. `(B1, B4)`.

Power pins (at the top or bottom bbox edge) exit vertically; signal pins alternate left/right.

---

## Placement mode

When the optimizer adds a `"placed"` key to a block (`x`, `y`, `rotation_deg`, `variant_index`), the viewer automatically switches from **Browser** to **Placed** mode and draws blocks at their optimizer-output coordinates.

## DRC check — in-app

Click the **DRC** toolbar button to open the DRC window.

```
┌─────────────────────────────────────────────────────────────┐
│  PDK Rules: […/gpdk090_device_rules.json]        [ Browse…] │
│  [ Run DRC ]                          27 violations found   │
│  Violations:                                                 │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ B0 <-> B1: different_bulks (active)  req 2µm  act 1µm│  │
│  │ B0 <-> B2: different_bulks (poly)    req 3µm  act 1µm│  │
│  │ …                                                     │  │
│  └───────────────────────────────────────────────────────┘  │
│  [ ] Show All Errors                      [ Clear Highlights]│
└─────────────────────────────────────────────────────────────┘
```

**Workflow:**

1. Click **Browse…** (or edit the path directly) to point to a PDK rules JSON.
2. Click **Run DRC** — the file is validated and all block pairs are checked.
3. Click any violation in the list → the gap between the two offending blocks is highlighted on the canvas.
4. Check **Show All Errors** → all violations are drawn simultaneously, each category in its own color.
5. Click **Clear Highlights** to remove overlays without closing the window.
6. Closing the DRC window removes all overlays automatically.

### Violation highlight colors

| Category | Color | Meaning |
|----------|-------|---------|
| `same_cellNames` | Yellow | Same device type, insufficient spacing |
| `same_deviceGroups` | Orange | Same voltage class, different cell |
| `different_cellNames` | Purple | Different cell types |
| `different_bulks` | Magenta | Different well/bulk connections |
| `different_deviceGroups` | Red | Different voltage class (LV ↔ HV) |
| Physical overlap | Red stripe | Bboxes physically intersect |

Each highlight is a filled rectangle drawn in the gap between the two blocks' closest facing edges, clipped to the shared band where those edges actually face each other.

## DRC evaluation — how it works

### PDK rules file

The checker reads `gpdk090_device_rules.json`, which contains three relevant sections:

**`device_topology`** — classifies each device type into a voltage group and a bulk connection:

```json
"device_topology": {
  "nmos1v_nat": { "device_group": "LV", "bulk": "VSSA" },
  "nmos2v_nat": { "device_group": "HV", "bulk": "VSSD" },
  "pmos1v_nat": { "device_group": "LV", "bulk": "VDDA" },
  "pmos2v_nat": { "device_group": "HV", "bulk": "VDDD" }
}
```

**`device_spacing`** — minimum spacing values (µm) for pairs of the **same** device type. Negative values mean a controlled overlap is allowed (e.g. merged diffusion):

```json
"nmos1v_nat": {
  "common_bulk":    { "active": -0.24, "poly": 1.0 },
  "common_active":  { "active": -0.24, "poly": 1.0 },
  "same_cellNames": { "active":  1.0,  "poly": 1.0 }
}
```

**`between_topologies`** — spacing costs for pairs of **different** device types, keyed by the topology category determined by the decision tree:

```json
"same_cellNames":         { "active": 2, "poly": 3 },
"same_deviceGroups":      { "active": 4, "poly": 5 },
"different_cellNames":    { "active": 6, "poly": 7 },
"different_bulks":        { "WPE_active": 2, "WPE_poly": 3 },
"different_deviceGroups": { "WPE_active": 5, "WPE_poly": 6 }
```

`WPE_active` / `WPE_poly` are Well-Proximity-Effect distances; they apply whenever devices cross a bulk or voltage-class boundary.

### Topology decision tree

For each block pair the checker walks this tree (read from the PDK, not hardcoded):

```
Are the device types the same?
  YES → device_spacing / same_cellNames
  NO  →
    Same device_group (LV/HV)?
      NO  → different_deviceGroups  (WPE rules)
      YES →
        Same bulk (VSSA/VDDA/…)?
          NO  → different_bulks      (WPE rules)
          YES → same_deviceGroups
```

Unknown device types fall back to `different_deviceGroups` (worst case).

### Spacing directions — active vs poly

Spacing rules are defined along two device-intrinsic axes, **not** absolute X/Y:

| Axis | Direction | Device rotation 0° | Device rotation 90° |
|------|-----------|--------------------|---------------------|
| `active` | Source–drain axis | Horizontal (X) | Vertical (Y) |
| `poly` | Channel-width axis | Vertical (Y) | Horizontal (X) |

### Facing-edge model

The checker uses the same **edge-projection model** as commercial DRC tools (Calibre, Virtuoso):

> A spacing rule between two shapes is checked **only** between edges that face each other — i.e. edges whose projections overlap.

For two axis-aligned bounding boxes:

- **Active check** fires when the blocks share a horizontal band (Y-extents touch or overlap). The gap measured is in X (active direction, rot=0).
- **Poly check** fires when the blocks share a vertical band (X-extents touch or overlap). The gap measured is in Y (poly direction, rot=0).
- **Diagonal pairs** (gaps in both X and Y simultaneously) have no facing edges and are skipped for non-WPE categories. For WPE categories (`different_bulks`, `different_deviceGroups`) a Euclidean corner-distance check is applied instead: `sqrt(active_gap² + poly_gap²)` must exceed `max(req_active, req_poly)`. This avoids false positives where one dimension has a large gap (e.g. 13 µm in Y) that makes the actual corner distance safe even though the other dimension's gap is small.

This avoids false positives for blocks that are side-by-side horizontally but happen to have zero vertical gap — a naive algorithm would report a poly violation of 0 µm for every horizontal neighbor.

### What is checked

All `N × (N−1) / 2` pairs of placed blocks are checked. For each pair:

1. **Physical overlap** — if the `main_bbox` rectangles intersect in both X and Y, a hard overlap error is reported and spacing checks are skipped.
2. **Active-direction spacing** — measured on the facing right/left edges (for rot=0). Violation if gap < required minimum.
3. **Poly-direction spacing** — measured on the facing top/bottom edges (for rot=0). Violation if gap < required minimum.
4. **Controlled overlap** — for negative required values (e.g. merged diffusion), a violation is only reported if the actual penetration depth exceeds the allowed limit.

All spacing values are in **micrometers (µm)** and reference the placed `main_bbox` coordinates from the optimizer output.

## Known limitations / future improvements

### Marginal spacing — boundary pairs

The optimizer sometimes places blocks at **exactly** the required spacing (gap == threshold). These are technically compliant but fragile — any floating-point rounding or PDK update would flip them into violations. The DRC checker currently treats them as clean (gap is not strictly less than threshold). A future improvement would be a separate "marginal" warning level that flags pairs within, say, 5 % of the required spacing.

### `same_deviceGroups` category currently unreachable

With the four device types in `gpdk090_device_rules.json` (`nmos1v`, `nmos2v`, `pmos1v`, `pmos2v`) every same-group pair (LV–LV or HV–HV) always has a different bulk, so all same-group pairs are classified as `different_bulks`. The `same_deviceGroups` spacing rules (active=4 µm, poly=5 µm) are never exercised. Once passive devices (RES, MIMCAP) are added to the PDK they will introduce pairs within the same group and same bulk, making this category active and its spacing rules testable.

### Mixed-rotation block pairs

When one block has rotation 0° and the other has rotation 90°, the checker defaults to treating both as 0°. The `active`/`poly` axes for the rotated block are swapped compared to what the PDK rules assume, so violations on the rotated block's poly axis may be missed or mis-reported. This has no impact while the optimizer only places blocks at 0°, but will need to be fixed once device rotation is enabled.
